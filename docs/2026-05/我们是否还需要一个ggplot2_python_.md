我们是否还需要一个ggplot2-python? by 生信编程自修室
------
<div><p data-pm-slice="0 0 []"><span leaf="">R 用户从 R 切回 Python 想画同样的图,默认选项是 </span><code><span leaf="">plotnine</span></code><span leaf="">。它把 Grammar of Graphics 在 Python 里实现得相当成熟,API 几乎和 R 端 ggplot2 逐字对应。但渲染那一头,plotnine 落在 matplotlib 上 —— 这是 Python 可视化生态的事实标准。问题不在 plotnine 写得好不好,而在 matplotlib 的 Figure / Axes / Artist 模型里,没有 R 端 grid 包提供的那一层 </span><em><span leaf="">abstract layout</span></em><span leaf=""> —— 没有可延迟解析的 Unit 表达式系统,没有 viewport 嵌套栈,没有 gtable 这种由网格化 grob 拼出整图的整体编排抽象。GoG 概念里像 panel、strip、legend 槽位、axis 边距、</span><code><span leaf="">theme(plot.background=...)</span></code><span leaf=""> 这些,在 R + grid 模型里是一阶对象,在 matplotlib 模型里需要靠 Axes / Figure 的属性堆叠加 </span><code><span leaf="">tight_layout</span></code><span leaf=""> / </span><code><span leaf="">subplots_adjust</span></code><span leaf=""> 这类启发式后处理。</span></p><p><code><span leaf="">Bio-Babel/ggplot2-python</span></code><span leaf=""> 这个新项目选了另一条路:不在 matplotlib 之上再贴一层,而是把 R 的整条底层栈(</span><code><span leaf="">grid</span></code><span leaf=""> / </span><code><span leaf="">gtable</span></code><span leaf=""> / </span><code><span leaf="">scales</span></code><span leaf="">)一起移植到 Python,然后让 ggplot 跑在 Python 端的 grid 上。代价是工作量翻了好几倍 —— 三个上游包都得自己端口;收益是 GoG 在 Python 里第一次有了和 R 端同构的分层抽象。</span></p><h2 align="center"><span leaf="">总览</span></h2><p><span leaf="">本文沿着仓库 </span><code><span leaf="">tutorials/geoms_gallery.ipynb</span></code><span leaf=""> 全部 cells 走一遍,先讲为什么 Python 还需要这个项目,再展示它在 5 个核心 geom(boxplot / violin / density / tile / hex)以及多层叠加上的 API 与 R 端 ggplot2 几乎一致。 重点是架构层面:</span><code><span leaf="">+</span></code><span leaf=""> 操作符、layer stack、scale 系统、coord 转换,最终如何落到 grid 上的 gtable 编排,共同构成了 GoG 在 Python 里第一次完整的 abstract layout 实现。</span></p><h2 align="center"><span leaf="">Grammar of Graphics 在 R 里为什么成立</span></h2><p><span leaf="">Grammar of Graphics 这个概念出自 Leland Wilkinson 1999 年的 </span><em><span leaf="">The Grammar of Graphics</span></em><span leaf="">(Springer)。Hadley Wickham 2005 年开始在 R 里把它实现成 ggplot2,2016 年的 </span><em><span leaf="">ggplot2: Elegant Graphics for Data Analysis</span></em><span leaf="">(Springer 2nd ed)是经典参考。GoG 的核心想法是:任何统计图形都能拆成几个正交组件 —— data、aesthetic mapping、geom、stat、coord、facet、scale、theme,它们能用 </span><code><span leaf="">+</span></code><span leaf=""> 任意组合。</span></p><p><span leaf="">R 端 ggplot2 之所以能把 GoG 落实成一个干净可扩展的库,关键是它</span><strong><span leaf="">没有自己造图形底层</span></strong><span leaf="">,而是建在 R 的三个底层包之上:</span></p><ul><li><code><span leaf="">grid</span></code><section><span leaf="">(R core 自带):一个抽象的二维布局系统。它提供 Unit(单位表达式:</span><code><span leaf="">npc</span></code><span leaf="">、</span><code><span leaf="">cm</span></code><span leaf="">、</span><code><span leaf="">lines</span></code><span leaf="">、</span><code><span leaf="">strwidth</span></code><span leaf=""> 等可以混合算术,</span><code><span leaf="">Unit(1, "npc") - Unit(2, "lines")</span></code><span leaf=""> 这种表达式在 viewport push 时延迟解析)、Viewport(可嵌套的视口,带局部坐标系)、Grob(图形对象树)。layout 不是函数式后处理,是一阶对象。</span></section></li><li><code><span leaf="">gtable</span></code><section><span leaf="">(Wickham 维护):基于 grid 的网格表格布局,把 grob 装进 row × col 的 cell。一个 ggplot 的最终产物本质上是一张 gtable —— title / subtitle 行、strip / panel / axis 行、legend 列、margin —— 全是 gtable cell。</span></section></li><li><code><span leaf="">scales</span></code><section><span leaf="">(Wickham 维护):统一抽象数据空间 ↔ 美学空间(色板、形状、大小)的训练 / 映射 / 反向映射,提供 breaks / labels / log10 / sqrt / reverse 等 transforms。</span></section></li></ul><p><span leaf="">ggplot2 在 R 里的代码量本身不算庞大,真正繁重的工作分担到了这三个上游包。GoG 的语法之所以"干净",是因为有这套底层抽象兜底。</span></p><h2 align="center"><span leaf="">Python 这边过去缺的是什么</span></h2><p><span leaf="">Python 之前缺的一直就是这套上游栈。matplotlib 不是它们的对应物 —— matplotlib 是一个完整的渲染引擎,但它的层级模型(Figure → Axes → Artist)里没有"延迟解析的 Unit 表达式"这种抽象,也没有 gtable 那种"整图 = 一张 cell 网格"的整体编排原语。</span></p><p><span leaf="">在这种基础上写一个 GoG 实现 —— </span><code><span leaf="">plotnine</span></code><span leaf="">(</span><code><span leaf="">has2k1/plotnine</span></code><span leaf="">)就是这条路
